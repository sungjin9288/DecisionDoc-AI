# DecisionDoc AI 정보보호 정책

## 1. 목적
DecisionDoc AI의 정보 자산을 보호하고 서비스 연속성을 유지한다.

## 2. 범위
본 정책은 DecisionDoc AI 플랫폼의 모든 구성 요소에 적용된다.

## 3. 보안 원칙
- **최소 권한**: 업무 수행에 필요한 최소한의 권한만 부여
- **심층 방어**: 다중 보안 레이어 적용
- **감사 추적**: 모든 중요 행위의 로그 기록 및 보존
- **암호화 우선**: 민감 데이터는 저장 및 전송 시 암호화

## 4. 접근 통제
- 사용자 인증: JWT 토큰 (8시간 만료)
- 역할 기반 접근: admin / member / viewer
- 로그인 실패: 15분 내 10회 초과 시 차단
- 세션: 30일 이후 재인증 필요

## 5. 암호화 기준
- 전송: TLS 1.2 이상
- 비밀번호: bcrypt (rounds=12)
- SSO 시크릿: Fernet authenticated encryption (PBKDF2 키 유도)
- JWT: HS256 (최소 32바이트 키)

## 6. 로그 보존
- 감사 로그: 최소 1년 보존
- 접근 로그: 90일 보존
- 로그 무결성: Append-only, 삭제/수정 불가

## 6.1 운영 권한/로그 정책 요약 (구현 기준)
- 배포 권한
  - AWS 경로는 GitHub OIDC deploy role로 `deploy` / `deploy-smoke` workflow만 허용
  - 운영자 CLI/콘솔은 진단·복구 전용 (무분별한 prod 재배포 금지)
- 런타임 인증 키
  - `DECISIONDOC_API_KEY` / `DECISIONDOC_API_KEYS` (API 인증)
  - `DECISIONDOC_OPS_KEY` (`/ops/*` 보호)
  - `/admin/tenants`를 포함한 admin endpoint는 인증된 admin JWT 또는 설정된 Ops key 중 하나를 요구한다. Ops UI는 공통 인증 header 조합을 사용해 로그인 세션을 보존하며 두 자격 증명을 동시에 요구하지 않는다.
  - Browser tenant header는 signed access token의 tenant claim과 동기화한다. Protected request는 signed tenant/user identity로 현재 tenant `UserStore`를 읽고 persisted role과 `is_active`를 authorization authority로 사용한다. Missing·inactive user의 기존 token은 `401`, 신뢰할 수 없는 user state는 `503`으로 fail closed 처리한다. Middleware public 예외인 `/events` query access token도 같은 검사를 거치며, user state가 전혀 없는 fresh install compatibility만 token payload를 사용한다. Login·register·refresh·LDAP login은 claims를 먼저 검증하고 access/refresh token과 tenant ID를 공통 commit helper로 저장한다. Tenant write를 포함한 저장이 실패하면 이전 token과 tenant를 복원하고 current user와 DocumentOps evidence를 바꾸지 않는다. 같은 tab의 동시 401은 하나의 refresh promise에 합류한다. Generic API request는 refresh 성공 뒤에도 자동 replay하지 않고 explicit retry를 요구하며, refresh 시작 뒤 현재 tab 또는 같은 origin의 다른 tab에서 auth credential·tenant storage가 바뀌면 늦은 응답을 `superseded`로 폐기한다. Unrelated storage key는 auth revision을 바꾸지 않는다. 다른 tab의 최종 signed user·tenant·role이 현재 page와 다르면 앱을 한 번 reload해 page-memory evidence를 새 authorization context에서 다시 구성하고, 같은 identity·role의 token rotation은 reload하지 않는다. Refresh credential이 400/401/403으로 거절된 경우에만 invalid-session cleanup을 수행하고, endpoint·response·storage failure는 기존 credential·tenant·작업 evidence를 보존한다. JWT tenant와 다른 selector 전환은 access preflight에서 거부하고 기존 tenant로 rollback하며, admin JWT도 `TENANT_MISMATCH`를 우회하지 않는다. 승인된 selector 전환도 tenant ID의 browser storage write가 성공한 뒤에만 in-memory context를 바꾼다. Storage 값 자체는 authorization 근거가 아니며 push invalidation과 cross-device browser storage coordination은 제공하지 않는다.
  - 위 same-identity·role token rotation 보존 규칙은 `credential_version`도 같을 때만 적용한다. Password change는 bcrypt hash와 version 증가를 같은 user-state CAS에 저장하고 이전 access/refresh token을 모두 `401`로 폐기한다. 변경 요청 browser는 응답의 새 pair를 공통 session helper로 commit하며, 다른 same-origin tab은 version mismatch에서 reload한다. Version 없는 legacy record/token은 persisted version이 0일 때만 호환한다.
  - 새 access/refresh pair는 tenant-scoped strict `auth-session.v2` object의 random session ID를 공유하고 refresh 뒤에도 같은 ID를 유지한다. 기존 v1 record는 exact-read compatibility를 유지하고 label mutation에서 v2로 승격한다. `/auth/logout`은 signed current session만 owner 조건 아래 CAS로 폐기해 별도 login을 유지한다. `GET /auth/sessions`는 현재 credential version의 본인 활성 session만 `no-store`로 반환하고, `PATCH /auth/sessions/label`은 최대 40자의 본인 기기 이름을 selected-backend CAS로 저장한다. API와 persisted state는 같은 Unicode-aware validator를 사용해 raw display control, line/paragraph separator, bidi/invisible format character를 trim 전에 거부하고 ZWNJ/ZWJ는 허용한다. Invalid persisted label은 원본 bytes를 rewrite하지 않고 fail closed 처리한다. `POST /auth/sessions/revoke`는 다른 로그인 하나를 exact owner 조건으로 종료하고, `POST /auth/sessions/revoke-others`는 strict confirmation 뒤 current를 제외한 검증 snapshot을 종료한다. `POST /auth/sessions/revoke-all`은 strict confirmation 뒤 현재 session을 마지막으로 써서 snapshot의 본인 active session 전체를 종료한다. Current selected target은 `409`, foreign/missing target은 같은 `404`이고 legacy sessionless token은 exact logout·inventory·label·selected/bulk revoke를 사용할 수 없다. Prefix의 모든 direct child는 mutation 전에 strict 검증되어 corrupt·unavailable state를 원본 보존 `503`으로 닫는다. Bulk write는 multi-object transaction이 아니므로 중간 failure의 부분 진행과 snapshot 뒤 새 session을 숨기지 않으며 재조회·재시도를 요구한다. Current-last 순서는 다른 session write 실패 시 현재 browser를 보존하지만 current write 응답 유실까지 원자적으로 조정하지는 않는다. Browser는 session ID를 DOM에 넣지 않고 label/revoke action의 늦은 응답을 같은 generation guard로 버리며, 전체 종료 성공 뒤에만 local credential과 미저장 evidence를 정리한다. Protected request, refresh exchange와 `/events`는 persisted session authority를 다시 확인한다. `GET /admin/auth-sessions/retention-preview`는 admin JWT 또는 Ops key만 허용하고 selected backend의 tenant prefix 전체를 strict 검증한다. `auth-session-retention-preview.v1`은 retention cutoff 이전 expired/revoked record를 식별자 없는 aggregate로만 반환하며 `read_only=true`, `deletion_authorized=false`, `no-store`를 고정한다. Audit에도 retention 기간과 inspected/eligible count만 남기고 token, user ID, session ID와 label을 복사하지 않는다. 손상 state는 원본 보존 `503`으로 닫고 조회 중 write/delete를 수행하지 않는다. User-Agent/IP, admin mass revoke와 실제 expired-session 삭제·scheduler는 제공하지 않는다. 일반 browser logout은 revoke 결과와 무관하게 local cleanup을 즉시 수행하고 endpoint failure를 서버 폐기 미확인 상태로 경고한다.
  - Ops retention panel은 access token 또는 Ops key가 있을 때만 30/90/180/365일 preview를 호출한다. Browser가 `auth-session-retention-preview.v1`, `read_only=true`, `deletion_authorized=false`, aggregate count와 timestamp consistency를 모두 검증한 최신 tenant/request 응답만 렌더링한다. Invalid contract는 fail closed error로 표시하고 식별자나 aggregate metric을 남기지 않는다. UI에는 삭제·scheduler·mutation control이 없다.
  - Retention policy comparison은 `GET /admin/auth-sessions/retention-comparison`의 한 번의 strict inspection에서 30/90/180/365일 aggregate를 함께 계산한다. `auth-session-retention-comparison.v1`은 user ID, session ID, label과 token을 노출하지 않고 `read_only=true`, `deletion_authorized=false`, `snapshot_atomic=false`, `requires_recheck_before_mutation=true`를 고정한다. Browser는 exact policy order, count 합계와 단조성, cutoff 시각을 검증하고 current tenant의 최신 응답만 사용한다. Selector는 검증된 응답 안에서만 이동하며 refresh만 새 inspection을 요청한다.
  - `GET /admin/auth-sessions/retention-handoff`는 admin JWT 또는 Ops key로만 접근하며 `retention_days`를 30/90/180/365로 제한한다. tenant-bound `auth-session-retention-review-handoff.v2`은 aggregate-only comparison과 canonical `comparison_sha256`만 포함하고 `review_only=true`, `policy_change_authorized=false`, `deletion_authorized=false`, `scheduler_authorized=false`, `snapshot_atomic=false`, `requires_recheck_before_mutation=true`, `handoff_persisted=false`를 고정한다. Response body SHA-256 header, `no-store`, `nosniff`, attachment filename을 함께 보내며 session object나 handoff artifact를 쓰지 않는다. v1은 historical evidence로 유지하지만 recheck source로는 거부한다. Browser는 body/header hash, comparison hash, authority flags와 current tenant/request generation을 모두 확인하기 전에는 화면 또는 download를 바꾸지 않는다. Handoff audit은 selected policy와 aggregate count, read-only boundary만 남긴다.
  - `POST /admin/auth-sessions/retention-handoff/recheck`는 admin JWT 또는 Ops key dependency를 body parsing보다 먼저 적용하고, exact `auth-session-retention-recheck-request.v1` body와 tenant-bound v2 source handoff의 canonical full-handoff SHA-256을 검증한다. Source/current handoff, handoff hash, aggregate fingerprint hash, tenant, selected policy와 false authority flags가 모두 맞을 때만 `auth-session-retention-recheck-receipt.v1` JSON attachment를 화면과 download에 사용한다. Receipt는 `aggregate_only=true`, `review_only=true`, `snapshot_atomic=false`, `requires_recheck_before_mutation=true`, `recheck_persisted=false`를 고정하며 source/current handoff, receipt, session, policy, scheduler state를 저장하지 않는다. `unchanged`는 aggregate equivalence일 뿐 session set identity나 mutation safety가 아니다. `changed`는 200 read-only result로 새 handoff가 필요하다는 뜻이며 승인이나 실행 권한을 만들지 않는다. Selector, refresh, tenant·auth-context change와 newer request는 page-memory source와 in-flight completion을 무효화한다. Audit `auth_session.retention_recheck`은 selected policy, current inspected/eligible count, aggregate status와 read-only/snapshot boundary만 남긴다.
  - `POST /admin/auth-sessions/retention-handoff/review-disposition`는 admin/Ops dependency가 malformed body validation보다 먼저 실행된다. Exact `auth-session-retention-review-disposition-request.v1`은 full H117 source receipt와 outer SHA-256, nested handoff SHA-256, aggregate fingerprint, tenant, selected policy, aggregate status, authority flags를 재검증한다. `auth-session-retention-review-disposition-receipt.v1`은 canonical `disposition_binding_sha256`, `receipt_status=issued`, `review_only=true`, `reviewer_identity_bound=false`, `approval_granted=false`, `execution_authorized=false`, `policy_change_authorized=false`, `deletion_authorized=false`, `scheduler_authorized=false`, `mass_revoke_authorized=false`, `snapshot_atomic=false`, `requires_recheck_before_mutation=true`, `decision_receipt_persisted=false`를 고정한다. unchanged에는 `acknowledged_unchanged|review_deferred`, changed에는 `new_handoff_required|review_deferred`만 허용한다. 인증 뒤 handler는 retention 대상 session/policy state를 조회·변경하지 않고 body SHA-256, `no-store`, `nosniff`, safe attachment을 반환한다. Audit `auth_session.retention_review_disposition`은 selected policy, aggregate status, disposition, source receipt SHA-256, emitted receipt SHA-256, review-only flag만 남기고 identity, session, IP, User-Agent, source body, token, key, label, free text를 남기지 않는다. UI는 verified recheck receipt만 page memory에 보관하고 policy/disposition selector, refresh, handoff, recheck, tenant/auth/Ops-key context와 newer request가 source와 늦은 completion을 무효화한다.
  - 열린 `/events` connection은 최대 15초마다 access token의 expiry와 현재 user/session authority를 다시 검사한다. Invalid credential은 private state가 없는 `auth_revoked` event 뒤 연결을 닫고, authority를 읽을 수 없으면 `auth_unavailable` event 뒤 application event delivery를 중단한다. Browser는 revoked event에서 refresh를 한 번만 수행하고 명시적으로 거절된 refresh credential만 destructive cleanup으로 연결한다. Authority·endpoint·storage failure는 credential과 미저장 evidence를 보존한 채 polling과 reconnect로 복구한다. 이는 exact-session과 current-inclusive bulk 폐기를 반영하는 bounded active-stream termination이며 즉시 cross-device push revocation이나 15초 이내 종료를 보장하지 않는다. 관리자 일괄 폐기와 expired-session delete/GC 실행도 현재 범위 밖이다.
  - DocumentOps 미저장 review draft는 사용자·tenant·trajectory page-memory key로만 유지하고 logout 또는 invalid session에서 폐기한다. localStorage와 server/audit에는 draft 본문을 저장하지 않는다.
- DocumentOps trajectory 상태
  - Redacted trajectory와 사람 review 이력은 local `data/tenants/<tenant_id>/trajectories.jsonl` 또는 같은 relative path의 S3 state object에 저장한다.
  - Tenant와 trajectory identity를 path·record에서 검증하고 blank·malformed·non-object JSONL, duplicate key·identity, non-finite number와 손상 private receipt는 조회·append·review를 중단하며 원본 bytes를 보존한다. 기존 tenant 미표기 record는 path-owned compatibility를 유지하고 explicit foreign record는 현재 tenant에 노출하거나 변경하지 않는다.
  - Append와 review는 local conditional file write 또는 S3 conditional create/ETag CAS로 확정하고 충돌마다 최신 record 집합에 operation을 최대 32회 재적용한다. Private append/incarnation identity와 최근 64개 review mutation receipt로 commit 응답 유실 뒤 successor mutation을 조정하며 public record와 SFT source에는 private metadata를 노출하지 않는다. Expected review version은 최신 CAS state에서 비교한다.
  - `trajectory_metadata.json`은 같은 selected backend의 별도 mutable CAS index다. SFT export, freeze, dry-run approval, execution request, pre-execution audit는 immutable conditional create 후 identity·size·SHA-256을 index에 결속하며 binding이 다른 export/audit bytes는 download하지 않는다. H74 이전 hash-only local record는 checksum 검증으로 읽되 size binding을 검증한 것으로 표시하지 않는다. Reviewer sign-off summary도 같은 backend prefix의 read-only handoff만 읽는다. Trajectory JSONL, metadata index와 여러 artifact를 함께 묶는 distributed transaction, metadata 확정 전 process 중단으로 남은 비권위 artifact의 자동 GC, 64개를 넘는 trajectory review reconciliation, retry backoff·fairness, 실제 AWS/provider/training runtime은 현재 보장 범위가 아니다.
  - Trajectory capture를 선택한 Agent run의 optional `operation_id`는 provider 호출 전에 `document_ops_agent_operations/<sha256>.json` private receipt로 선점한다. Receipt에는 request hash만 저장하고 tenant·operation·owner·상태·UTC timestamp를 strict schema로 검증한다. 성공 상태는 저장된 trajectory-bound result와 hash를 포함하며 exact replay만 반환한다. Changed payload, running/failed retry와 손상 receipt는 provider 호출 전에 각각 `409` 또는 `503`으로 중단한다. API-key 보호 status 조회는 current tenant path만 읽고 operation ID·상태·시각·다음 행동만 반환한다. Owner, request/result hash와 결과 본문은 응답·audit에서 제외하며 missing tenant path는 `404`, 손상 state는 `503`으로 닫는다. Browser는 status를 cache하지 않고 schema·operation identity·state별 필드·read-only·provider-call 비승인을 모두 검증한다. Mismatched·unavailable·running 상태는 captured tenant와 payload를 page memory에만 보존해 두 recovery control이 새 operation 대신 같은 status를 single-flight로 다시 읽는다. Captured POST 직전에는 payload를 제외한 schema·tenant·operation ID만 tenant-scoped same-origin marker로 기록하고, 지원되는 browser에서는 tenant별 Web Lock으로 claim을 직렬화한다. 다른 tab이 marker를 먼저 확정하면 새 POST 없이 기존 status를 읽는다. Marker key를 tenant별로 분리해 foreign tenant read/write/clear가 다른 tenant marker를 삭제하지 않으며, 승인된 tenant 전환은 이전 tenant marker만 제거한다. H96 base-key marker는 owning tenant만 호환해 읽고 제거한다. Reload·tab close 뒤에도 strict marker와 current tenant가 일치할 때 status-only read로 새 POST를 막고, backend 실행 비취소와 evidence 확인 경고를 승인한 명시적 release에서만 marker를 제거한다. Tenant-scoped slot의 extra field, wrong tenant, wrong schema, non-browser UUID marker는 제거하고 logout·invalid session도 현재 context marker를 폐기한다. Shared storage 접근 실패는 tenant-scoped tab fallback으로 내려가고 두 storage 접근 실패도 same-page 실행을 막지 않는다. Terminal success의 exact replay는 payload가 남은 같은 page에서만 가능하다. Marker와 Web Lock은 authorization primitive가 아니며 다른 browser/device, Web Locks 미지원 환경의 완전 동시 claim, process crash 자동 복구, 서로 다른 ID의 의미상 중복 차단, exactly-once provider 실행, receipt expiry·GC는 제공하지 않는다.
- Root tenant registry 상태
  - Tenant record와 API key hash는 local `data/tenants.json` 또는 같은 relative path의 S3 state object에 저장한다. Mutation receipt는 public tenant identifier로 유효하지 않은 private root sentinel 아래에 두고 admin/API 응답에서는 제거한다.
  - Target mutation은 tenant identity와 record schema를 다시 검증하며 malformed private metadata, foreign ownership과 invalid target record는 조회·변경 또는 인증 성공으로 축소하지 않고 원본을 보존한다.
  - Create/update/hint/API key rotation은 local conditional file write 또는 S3 conditional create/ETag CAS로 확정하고 충돌마다 최신 target에 operation을 최대 32회 재적용한다. 최근 64개 private mutation receipt로 commit 응답 유실 뒤 successor mutation을 조정한다.
  - API key rotation은 한 번 생성한 plaintext와 hash를 같은 mutation에 결속한다. Lost response는 현재 hash가 정확히 하나의 active owned tenant에 결속돼 실제 인증 가능한 경우에만 성공으로 조정하며, 후속 rotation·비활성화·foreign ownership·duplicate hash는 fail closed 처리한다. 이 보장은 단일 root registry object 범위이고 실제 AWS runtime과 다른 state object를 묶는 transaction은 현재 보장 범위가 아니다.
- 계정·초대 상태

  - Admin update는 strict Pydantic schema를 사용하고 masked secret 재전송은 기존 암호문을 유지한다. GCloud state와 SAML RelayState는 constant-time 비교하며 SAML ACS는 IdP certificate와 signed assertion 검증을 요구한다. Verifier가 없으면 인증을 거부한다.
  - Save/partial update는 local conditional file write 또는 S3 conditional create/ETag CAS로 확정하고 충돌마다 최신 owned config에 변경을 최대 32회 재적용한다. 최근 64개 private mutation receipt는 public config에서 제거하고 commit 응답 유실 뒤 successor update를 조정한다. Process-local lock은 contention 완화 수단이며 persistence authority가 아니다. 이 보장은 단일 SSO config object 범위이고 실제 LDAP/SAML/GCloud 로그인 성공은 현재 보장 범위가 아니다.
- 스타일 프로필 상태
  - tone guide, bundle override, 분석 예시와 default/system metadata는 local `data/tenants/<tenant_id>/style_profiles.json` 또는 같은 relative path의 S3 state object에 저장한다.
  - tenant와 owned profile의 exact schema를 state 접근 전에 검증한다. Malformed JSON, duplicate key, identity/timestamp drift, duplicate example ID와 multiple default는 조회·prompt build와 후속 변경을 중단하며 원본 bytes를 보존한다. Explicit foreign record는 현재 tenant에 노출하거나 변경하지 않고 보존한다.
  - Profile mutation은 local conditional file write 또는 S3 conditional create/ETag CAS로 확정하고 충돌마다 최신 ownership·schema·profile identity 위에 최대 32회 재적용한다. Private incarnation은 같은 public profile ID의 replacement lifecycle을 구분하고 최근 64개 mutation receipt와 함께 public profile reader에서 제거한다. Process-local lock은 contention 완화 수단이며 persistence authority가 아니다. 이 보장은 단일 style profile object 범위이고 실제 provider 기반 style analysis 성공은 현재 보장 범위가 아니다.
- 품질 학습 상태
  - 사용자 feedback, eval evidence, runtime prompt override, A/B prompt experiment와 request pattern은 local `data/tenants/<tenant_id>/{feedback.jsonl,eval_results.jsonl,prompt_overrides.json,ab_tests.json,request_patterns.jsonl}` 또는 같은 relative path의 S3 state object에 저장한다.
  - Tenant와 owned record schema를 state 접근 전에 검증한다. Malformed/invalid UTF-8 JSON/JSONL, duplicate key·identity, identity/type/timestamp/score drift는 dashboard·eval·feedback·A/B·request-pattern API와 생성 prompt build, 후속 변경을 중단하며 원본 bytes를 보존한다. Explicit foreign record는 현재 tenant에 노출하거나 변경하지 않고 보존하며 tenant 필드 없는 기존 record는 path ownership으로 읽는다.
  - Runtime prompt override, A/B experiment, request pattern mutation은 local conditional write 또는 S3 conditional create/ETag CAS로 확정한다. 충돌마다 최신 ownership·schema 위에 operation을 최대 32회 재적용한다. Override refresh는 같은 incarnation과 applied count를 유지하고 payload-bound save receipt로 동일 operation ID의 다른 payload를 거부한다. Incarnation 필드가 없는 기존 override는 bundle·생성 시각·tenant binding에서 deterministic lineage를 계산해 worker마다 다른 생명주기로 갈라지지 않게 한다. Private mutation metadata는 public 응답에서 제거한다. Request clear는 최초 snapshot의 unmatched identity만 제거해 후속 append를 보존한다.
  - A/B assignment는 variant·hint·experiment identity를 한 CAS에서 확정하고 result와 conclusion을 같은 identity에만 적용한다. Pending conclusion은 persisted sample·winner score·hint·mutation receipt와 일치해야 하며, 같은 operation ID로 winner prompt override를 저장한 뒤 experiment를 concluded로 기록한다. Override 저장 실패는 public active experiment와 pending claim을 유지하고 오류를 상위 호출자에 전달하며 다음 evaluation이 같은 claim을 재개한다. Pending claim 동안 create·result는 기존 experiment를 바꾸지 않고 reset은 `409` conflict로 거부한다.
  - Fine-tune dataset/export와 model registry도 같은 tenant의 local/S3 state object에 저장한다. Dataset, export metadata/content와 model lifecycle schema를 fail closed로 검증하고 export size/SHA-256, request/model/provider-job/private mutation identity를 중복 없이 결속한다. Dataset append·clear, export metadata append와 model lifecycle mutation은 객체별 conditional create/CAS로 확정하고 private append identity·immutable model incarnation·bounded receipt는 public 응답에서 제거한다. Metadata가 확정되지 않은 export orphan은 download/upload authority로 사용하지 않는다. 손상된 registry를 active model 없음으로 축소하지 않으며 provider job 성공 모델은 promotion eval을 마치기 전까지 inactive로 유지한다.
  - 자동 provider training은 기본값 `FINETUNE_AUTO_ENABLED=0`이고 opt-in과 threshold를 모두 만족해야 한다. Orchestrator는 명시적 execution authority가 없는 호출에서 dataset upload와 provider job creation을 수행하지 않는다.
  - Feedback/eval append, prompt override, A/B experiment, request pattern, fine-tune dataset/export metadata와 model registry의 persistence authority는 객체별 conditional CAS이며 process-local lock은 contention 완화 수단이다. Feedback/eval은 기존 JSONL prefix를 보존하고 stable append identity로 불확실한 commit을 조정하며 eval private identity는 public record에 노출하지 않는다. Fine-tune export content는 immutable conditional create지만 metadata와 함께 묶는 distributed transaction은 아니다. 각 보장은 단일 state object 범위이며 실제 AWS/provider API, dataset upload, training execution, external job polling과 model promotion은 별도 운영 승인 범위다.
- 공개 공유 상태
  - share link는 local `data/tenants/<tenant_id>/shares.json` 또는 같은 relative path의 S3 state object에 저장한다.
  - tenant를 path 선택 전에 검증하고 blank·malformed·non-object JSON, duplicate key, owned malformed record, storage key/share ID drift와 손상 private receipt는 공개 조회와 후속 생성·접근 횟수·취소 변경을 중단한다. Explicit foreign record는 현재 tenant에 노출하거나 변경하지 않고 보존하며 기존 tenant 미표기 record는 path ownership으로 읽는다.
  - Worker mutation은 local conditional file write 또는 S3 conditional create/ETag CAS로 확정하고 충돌마다 최신 ownership·schema·lifecycle 위에 create/access/revoke를 재적용한다. 최근 mutation receipt는 64개로 제한하고 public 응답에서는 제거하며 최초 취소자와 시각을 보존한다. 이 보장은 단일 share state object 범위이며 실제 AWS runtime과 운영 URL의 외부 접근성은 현재 보장 범위가 아니다.
- 감사 로그 저장
  - 저장 위치: local `data/tenants/<tenant_id>/audit_logs.jsonl` 또는 같은 relative path의 S3 state object
  - tenant와 log identity를 append 전에 검증하고 기존 JSONL byte prefix를 보존한다. 손상·foreign·중복 evidence는 자동 복구하거나 건너뛰지 않고 read와 append를 중단한다.
  - 독립 worker의 append는 missing object의 conditional create와 기존 object의 validated-prefix CAS로 확정한다. 충돌하면 최신 JSONL을 다시 검증하고 최대 32회 재시도하며, commit 응답 유실 뒤 successor append가 발생해도 `log_id`와 exact entry read-back으로 성공을 조정한다. 실제 AWS runtime은 현재 검증 범위가 아니다.
  - 조회/내보내기: `GET /admin/audit-logs`, `GET /admin/audit-logs/export`
  - 조회와 CSV 내보내기는 tenant와 action/result/기간 filter를 공유한다. 조회는 검증된 offset/limit과 전체 건수·다음 페이지 여부를 반환하고, CSV는 전체 detail JSON을 보존하며 spreadsheet formula injection이 가능한 문자열을 text cell로 처리
