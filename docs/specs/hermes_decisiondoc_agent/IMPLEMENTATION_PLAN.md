# Hermes-Inspired DecisionDoc Agent Implementation Plan

## Scope

Implement a DecisionDoc-native DocumentOps and PolicyOps AI path inspired by Hermes Agent.

This is not a direct Hermes runtime import. It is an additive DecisionDoc feature path that keeps current project, knowledge, generation, approval, export, audit, provider, and fine-tune systems intact.

## Non-Goals

- Do not add Hermes as a production dependency yet.
- Do not enable Hermes terminal/browser/remote execution tools in production.
- Do not train a model before curated reviewed data exists.
- Do not bypass DecisionDoc providers.
- Do not break `mock` provider behavior.
- Do not send sensitive documents to external APIs without explicit provider/tenant configuration.

## Phase 5: Reviewed Dataset Inspection

Status: implemented for API/store/UI dry-run preview; no model training.

Purpose:

- Inspect reviewed trajectory quality before writing SFT JSONL.
- Report eligible and blocked records.
- Make blocked reasons visible before operators create an export.
- Keep export generation explicit and ops-key protected.

Implemented behavior:

- `TrajectoryStore.preview_sft_export()` performs a dry run without writing files.
- Preview reports candidate count, eligible count, blocked count, blocker summary, quality score summary, task/skill counts, and sample records.
- SFT export skips accepted records that are structurally unsafe for SFT export.
- `/api/agent/document-ops/trajectories/export/preview` exposes the dry run behind the ops key gate.
- The DocumentOps UI exposes an `Export preview` action before `SFT export`.

## Phase 6: Secure Reviewed SFT Artifact Access

- `/api/agent/document-ops/trajectories/reviewed-sft-exports/{filename}/download` downloads only generated `accepted_only=true` reviewed SFT JSONL artifacts behind the ops-key gate.
- Export downloads are constrained to safe SFT JSONL filenames and resolved paths inside the tenant export directory.
- The DocumentOps UI exposes a `Reviewed JSONL` action and per-artifact `다운로드` action using the reviewed-only endpoints.

## Phase 13: Two-Person Training Execution Request Record

- Requests reject `start_training=true`, `upload_dataset=true`, `call_provider_api=true`, and same-person requester/approver combinations.

## Phase 22: Local Reviewer Sign-Off Record Validator

- Ensure required reviewer roles, names, teams/titles, timestamps, decisions, evidence lists, notes, and acknowledgements are complete.
- Keep protected no-training boundary flags false even when actual reviewer sign-off is recorded outside the template.

Implemented behavior:

- `docs/specs/hermes_decisiondoc_agent/phase21_reviewer_signoff/validate_signoff_record.py` validates completed sign-off records from the command line and returns JSON output plus a process exit code.
- Completed records must include Product/PM, ML/AI owner, Compliance/Security, and Release owner reviewer roles.
- Each required reviewer must have a non-empty name, team/title, ISO 8601 timestamp, non-pending allowed decision, non-empty evidence list, and all required acknowledgements checked.
- `changes_requested` and `blocked` decisions require notes.
- Protected boundary flags for training execution, dataset upload, provider fine-tune API calls, provider job creation/polling, model candidate emission, and model promotion must remain `false`.
- The pending Phase 21 template is expected to fail completed-record validation until humans fill a copy.
