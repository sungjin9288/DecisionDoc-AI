## Current DecisionDoc Fit

DecisionDoc already has the following related rails:

- Attachment parsing in `app/services/attachment_service.py`
  - Direct extraction for PDF, DOCX, PPTX, HWPX, Excel, CSV/TSV, JSON/YAML/XML/HTML/RTF, OpenDocument, ZIP.
  - Per-file and global character caps.
  - Provider fallback for image files and scanned PDFs.
- MarkItDown local harness in `scripts/markitdown_ingest.py` and `scripts/run_ingestion_harness.py`
  - Currently optional integration dependency, not production runtime dependency.
- Project knowledge storage in `app/storage/knowledge_store.py`
  - Project-scoped document store, quality tiers, learning modes, bundle applicability, success-state ranking.
- Report Workflow ERP in `app/services/report_workflow_service.py`
  - Planning, slide generation, final approval, PPTX export, Project/Knowledge promotion.

## Integration Candidates

### 1. MarkItDown Runtime Adapter

Priority: High

What to import conceptually:
- Use MarkItDown as a normalized fallback converter for formats where our local parser is weaker.
- Prefer the narrow `convert_stream()` path for uploaded bytes rather than passing arbitrary paths or URLs.
- Keep plugins disabled by default.
- Make it optional behind an env flag and optional dependency group.

Why it fits:
- DecisionDoc's core user pain is mixed RFP/proposal input: PDF, DOCX, PPTX, XLSX, ZIP, images.
- We already document MarkItDown ingestion, but the UI upload path still relies on our local parser first.
- MarkItDown's Markdown output is more aligned with LLM prompt structure than plain text extraction.

Proposed design:
- Add `app/services/markitdown_adapter.py`.
- Add config:
  - `DECISIONDOC_MARKITDOWN_ENABLED=0`
  - `DECISIONDOC_MARKITDOWN_PLUGINS_ENABLED=0`
  - `DECISIONDOC_MARKITDOWN_MAX_CHARS=12000`
- Add a safe adapter function:
  - `convert_upload_to_markdown(filename: str, raw: bytes) -> str`
  - Only accepts bytes already uploaded through DecisionDoc.
  - Uses in-memory stream plus extension metadata.
  - Rejects URL-like filenames and remote fetches.
- Wire into `attachment_service.extract_text_with_ai_fallback()` after local parser failure, before provider OCR fallback for supported office formats.

Guardrails:
- Do not use MarkItDown's broad `convert()` entrypoint for server uploads.
- Do not enable remote URI conversion.
- Do not enable plugins in production by default.
- Preserve existing 20 MB upload cap and text caps.

MVP test cases:
- DOCX/PPTX/XLSX parser failure falls back to adapter when enabled.
- Adapter is not called when disabled.
- URL-like filename is rejected or treated as plain filename, never fetched.
- Plugin flag default is false.
- Existing `/generate/from-documents` tests still pass.

### 2. MemPalace-Inspired Knowledge Retrieval Upgrade

Priority: High, but implement as internal design rather than vendor dependency first.

What to import conceptually:
- Keep verbatim source text, do not summarize destructively.
- Split knowledge into scoped areas:
  - organization/client
  - project
  - report workflow
  - topic/tag
  - approved output
- Add hybrid ranking:
  - current metadata score
  - keyword overlap
  - temporal recency/proximity
  - approved/awarded success state
  - optional vector backend later
- Add temporal relationship graph for facts that change over time:
  - client preferences
  - evaluator requirements
  - project constraints
  - PM/대표 approval decisions

Why it fits:
- DecisionDoc currently has useful metadata ranking, but no vector retrieval and no temporal entity graph.
- Report Workflow ERP will produce many approved planning and slide artifacts. Without better retrieval, Knowledge will become a flat store.
- MemPalace's "local-first, opt-in, scoped retrieval" maps well to DecisionDoc's tenant/project boundaries.

Proposed phase 1:
- Add a `KnowledgeScope` model internally:
  - `tenant_id`
  - `project_id`
  - `organization`
  - `report_workflow_id`
  - `bundle_type`
  - `topic_tags`
- Extend `KnowledgeStore.rank_documents_for_context()` with:
  - recency score
  - workflow source score
  - query term density
  - selected reference explanation
- Add a UI panel showing "왜 이 참고자료가 선택됐는지".

Implementation status:
- `2026-04-27`: Phase 1 metadata scope was added as `knowledge_scope.v1` on knowledge entries.
- Current scope fields are `project_id`, `organization`, `report_workflow_id`, `bundle_types`, `topic_tags`, and source metadata.
- Tenant-level scope remains enforced by the existing API/storage boundary rather than duplicated inside `KnowledgeStore`.
- `2026-04-27`: Context preview observability was added through `knowledge_context_preview.v1`.
- `/knowledge/{project_id}/context` now returns `applied_scope` and `ranking_summary` so operators can verify which scope filters were applied and how many references matched bundle, organization, and workflow signals.
- The admin Knowledge context preview modal now exposes scope filters and displays server-applied filters plus ranking summary counts.

Proposed phase 2:
- Add optional SQLite FTS5 or vector backend abstraction:
  - `KnowledgeSearchBackend`
  - `LocalKeywordBackend`
  - later `VectorBackend`
- Keep default storage local and opt-in.

Implementation status:
- `2026-04-27`: `KnowledgeSearchBackend` and deterministic `LocalKeywordBackend` were added as the phase 2 seam.
- `KnowledgeStore.rank_documents_for_context()` now delegates keyword overlap calculation to the local backend and returns `search_backend`, `query_terms`, and `matched_query_terms` for explainability.
- `2026-04-28`: Optional `SQLiteFtsBackend` was added behind `DECISIONDOC_KNOWLEDGE_SEARCH_BACKEND=sqlite_fts`.
- The default remains `local_keyword`; SQLite FTS5 uses Python stdlib `sqlite3`, introduces no production dependency, and falls back to local keyword matching when FTS5 is unavailable.
- Vector search remains a future optional backend.

Proposed phase 3:
- Add temporal knowledge graph:
  - entities: organization, project, person/role, requirement, artifact
  - relations: `requires`, `approved`, `requested_change`, `uses_template`, `won_with`
  - validity windows: `valid_from`, `valid_to`

Implementation status:
- `2026-04-27`: Phase 3 read-only temporal graph MVP was added as `knowledge_temporal_graph.v1`.
- `KnowledgeStore.build_temporal_graph()` now derives project, artifact, organization, report workflow, bundle, topic, and reuse-success relationships from normalized Knowledge metadata.
- `GET /knowledge/{project_id}/temporal-graph` exposes the graph with the same bundle, organization, and workflow filters used by context preview.
- This remains dependency-free and read-only; person/role, requirement-level relations, and mutation APIs are intentionally deferred.
- `2026-04-28`: Relationship graph evidence is now reflected in Knowledge ranking with a capped `graph_relationship_score`.
- The boost is intentionally conservative: generic `contains_artifact` relations do not add score, while workflow, bundle, organization, topic, approved reuse, and awarded reuse relations add a bounded signal exposed through context preview and applied reference metadata.

MVP test cases:
- Approved outputs outrank draft references.
- Same project and same bundle references outrank unrelated references.
- Recency does not override `gold/awarded` quality by itself.
- Selection reason is returned with every selected document.
- Tenant/project boundary is preserved.

### 3. ima2-gen Inspired Visual Asset Workspace

Priority: Medium

What to import conceptually:
- Node-style visual exploration for report graphics and cover images.
- Reference image limits and compression hints.
- Style sheet capture and reuse.
- Observable job list with request IDs and safe logs.
- Session/local gallery concept for generated visual assets.

Why it fits:
- DecisionDoc Report Workflow already plans slide layout and visual direction, but it does not manage visual assets as first-class artifacts.
- Proposal decks often need cover images, concept diagrams, architecture diagrams, icon sets, and section dividers.
- A lightweight "visual asset workspace" would improve report quality without replacing the current PPTX export.

Do not import directly:
- Do not copy OAuth-based image generation path. DecisionDoc production already uses provider routing and API-key based backend execution.
- Do not adopt its Node/React frontend wholesale; DecisionDoc currently uses a single static admin UI.

Proposed MVP:
- Add "Visual Assets" section inside Report Workflow detail.
- For each slide draft:
  - `visual_prompt`
  - `reference_refs`
  - `generated_asset_ids`
  - `selected_asset_id`
- Add backend model/store fields first; UI can show stub controls before actual image generation.
- Reuse existing provider capability route `visual=openai` where available.

MVP test cases:
- Visual asset metadata can be attached to a slide without changing approval state.
- Final approval locks selected visual asset metadata.
- PPTX export can include selected asset metadata initially as speaker note or placeholder.

## Recommended Implementation Order

1. MarkItDown safe runtime adapter
   - Smallest direct quality gain.
   - Uses existing docs and optional dependency.
   - Improves `/generate/from-documents` and Report Workflow source material quality.

2. Knowledge ranking explanation and scope model
   - Builds on existing `KnowledgeStore`.
   - No new external runtime service required.
   - Directly improves reuse of approved Report Workflow outputs.

3. Report Workflow visual asset metadata
   - Adds structure before adding expensive image generation.
   - Enables future image provider work without overloading current PPTX export.

4. Optional vector/FTS backend
   - Only after the metadata ranking UI proves valuable.
   - Keep backend pluggable and local-first.

## Decision

Proceed with MarkItDown adapter first. It is the most directly useful and lowest architectural risk because DecisionDoc already has an optional MarkItDown harness and a clear attachment ingestion seam.
