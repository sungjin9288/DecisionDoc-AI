## Integration Decisions

### 1. Editable PPTX Quality Contract

Priority: High

Source inspiration:
- `ppt-master` emphasizes native editable PPTX output: real PowerPoint text boxes, shapes, charts, and Office-compatible fallbacks.
- DecisionDoc already exports PPTX through `python-pptx`, so the safest integration is to strengthen our own prompt and export adapter rather than importing external code.

DecisionDoc implementation direction:
- Planning prompt must ask for PowerPoint-safe layout, visual direction, required evidence, design notes, and slide acceptance criteria.
- Slide generation prompt must preserve the approved planning snapshot and produce concise copy that can become editable shapes.
- PPTX export adapter should map planning metadata into `slide_outline` fields:
  - `core_message`
  - `evidence_points`
  - `visual_type`
  - `visual_brief`
  - `layout_hint`
  - `decision_question`
  - `acceptance_criteria`
- PPTX renderer should render decision questions and acceptance criteria as native text/cards, not image-only content.

Guardrails:
- Do not copy `ppt-master` workflow prompts or scripts without a separate file-level review.
- Do not introduce SVG conversion dependencies in production until a dedicated adapter and tests exist.
- Keep the current `python-pptx` path dependency-light and deterministic.

### 2. Browser Slide Editor UX

Priority: Medium, reference only

Source inspiration:
- `PPTist` has a mature browser slide editing model.
- `banana-slides` has AI slide generation UX and provider split.

DecisionDoc implementation direction:
- Use these repositories only as clean-room UX references for a later `Slide Studio`.
- Near-term UI should remain inside the existing admin static page and focus on staged approval, slide-level revision, asset selection, and PPTX export.

Guardrails:
- Do not copy AGPL source code, components, styles, or data models.
- If a full slide editor becomes necessary, design our own data model and component tree.

### 3. Knowledge Graph and Agent Skill Governance

Priority: Medium

Source inspiration:
- `graphify` validates the value of portable graph JSON, ignore rules, graph reports, and optional MCP/Neo4j exports.
- `openai/skills`, `everything-claude-code`, and `antigravity-awesome-skills` validate skill packaging, validation, metadata sync, and risk labeling.

DecisionDoc implementation direction:
- Keep current Knowledge graph dependency-free in runtime.
- Add future optional offline graph ingestion/export instead of adding heavy tree-sitter/NetworkX dependencies to production.
- Add a small DecisionDoc skill registry validator later for `.agents/skills`.

Guardrails:
- Do not import third-party skill instructions wholesale.
- Treat skill repositories as operational patterns, not trusted runtime content.

## Recommended Implementation Order

1. Strengthen Report Workflow PPTX generation with an editable PPTX contract.
2. Add external repo analysis documentation and link license decisions.
3. Add optional future spec for SVG-to-PPTX adapter only after current PPTX renderer exposes enough slide semantics.
4. Add DecisionDoc skill registry validator if `.agents/skills` maintenance starts creating drift.
5. Keep AGPL projects as non-code references unless a commercial license or compliance decision is made.
