# DecisionDoc AI 대표 시연 Runbook

## 1. 목적

이 문서는 외부 회사 대표, 의사결정권자, 초기 파일럿 검토 대상에게 DecisionDoc AI를 설명하고 시연할 때 사용하는 운영자용 스크립트입니다.

핵심 목적은 아래 3가지를 짧은 시간 안에 명확하게 보여주는 것입니다.

- 이 제품이 단순 요약 도구가 아니라 실제 문서 운영 플랫폼이라는 점
- 내부 설치형 구조로 고객사별 분리 운영이 가능하다는 점
- 실제 문서 생성과 내보내기까지 바로 가능한 상태라는 점

## 2. 현재 기준 시연 환경

### 접속 주소

- `https://admin.decisiondoc.kr`

### 현재 검증 완료 범위

아래 항목은 현재 `admin` 환경에서 실제로 검증된 범위입니다.

- HTTPS 접속 정상
- `/health` 정상 응답
- 인증 없는 생성 요청 차단
- 인증 포함 문서 생성 성공
- 인증 포함 export 성공

### 선택 시연 범위

아래 항목은 제품 차별화 설명에는 중요하지만, 시연 전 별도 데이터 준비 여부를 확인하고 보여주는 것을 권장합니다.

- 프로젝트/워크스페이스 흐름
- 결재/공유/이력 화면
- Public Procurement Go/No-Go Copilot
- G2B 연동 시나리오

## 3. 시연 전 체크리스트

### 운영 체크

- `https://admin.decisiondoc.kr/health` 응답 확인
- 브라우저에서 `admin.decisiondoc.kr` 접속 확인
- 운영자 계정 로그인 상태 확인
- 소개용 PDF 파일 준비
  - `output/pdf/decisiondoc_ai_executive_intro_ko.pdf`

### 말하기 준비

- 첫 1분 설명은 제품 포지셔닝만 전달
- 데모 중에는 “이 제품이 어떤 조직 문제를 줄이는가”에 집중
- 기능 나열보다 “문서 생성 → 운영 → 제출” 흐름으로 설명

### 피해야 할 것

- 아직 별도 고객 환경이 없는 기능을 실제 운영 완료처럼 말하지 않기
- 준비되지 않은 live G2B 공고를 즉석에서 검색해서 시연하지 않기
- 내부 설정값이나 API 키 화면을 노출하지 않기

## 4. 권장 시연 흐름

### 4.1 1분 포지셔닝

아래 문구로 시작하는 것을 권장합니다.

`DecisionDoc AI는 자료를 읽고 정리하는 도구가 아니라, 조직이 실제 문서를 만들고 검토하고 승인하고 제출하는 내부 설치형 문서 운영 플랫폼입니다.`

이후 소개용 PDF 첫 페이지를 열고 아래만 짧게 짚습니다.

- 어떤 문제를 해결하는지
- 왜 내부 설치형이 중요한지
- 왜 일반 요약형 AI와 포지션이 다른지

### 4.2 3분 핵심 데모

핵심 데모는 현재 검증된 범위 중심으로 진행합니다.

1. `admin.decisiondoc.kr` 접속
2. 문서 생성 입력 화면 또는 문서 생성 흐름으로 진입
3. 문서 제목, 목표, 컨텍스트를 짧게 입력
4. 생성 실행
5. 결과 문서가 생성되는 흐름 설명
6. export 결과가 실제 제출 가능한 형식으로 나간다는 점 설명

여기서 강조할 포인트:

- 단순 답변이 아니라 문서 산출물 중심이라는 점
- 결과를 바로 업무 파일로 넘길 수 있다는 점
- 운영 흐름 안에서 재사용 가능한 구조라는 점

### 4.3 2분 운영 구조 설명

시연 후에는 화면보다 구조를 설명합니다.

- `admin.decisiondoc.kr` 는 공용 운영/데모 환경
- 실제 고객 도입 시에는 고객사 전용 서브도메인으로 분리
- 키, 데이터, 볼륨, 운영 로그를 환경별로 분리 가능

권장 설명 문구:

`지금 보시는 환경은 공용 데모/운영 환경이고, 실제 도입 시에는 귀사 전용 환경으로 독립 배포합니다. 따라서 다른 고객 데이터와 섞이지 않게 운영할 수 있습니다.`

### 4.4 2분 차별점 설명

NotebookLM이나 일반 생성형 AI와 비교 질문이 나오면 아래 순서로 답합니다.

1. NotebookLM은 읽기와 이해에 강함
2. DecisionDoc AI는 작성과 운영에 강함
3. 내부 설치, 승인, 공유, 제출형 export, 고객별 분리 운영이 차이점

짧은 답변:

`NotebookLM이 자료 이해 도구라면, DecisionDoc AI는 실제 문서를 만들고 운영하는 플랫폼입니다.`

## 5. 선택 시연 시나리오

### 5.1 프로젝트/문서 운영 시나리오

아래 상황에서 보여주기 좋습니다.

- 문서 이력 관리가 중요한 고객
- 팀 단위 협업을 보는 고객
- 생성 이후 운영까지 궁금해하는 고객

설명 포인트:

- 프로젝트 단위로 문서를 묶을 수 있음
- 문서 생성 이후 검토/공유/이력 관리 가능
- 동일 프로젝트 안에서 후속 문서를 이어서 생성 가능

### 5.2 Public Procurement Go/No-Go 시나리오

아래 상황에서만 권장합니다.

- 공공조달/입찰 고객
- G2B와 제안서 문서화 흐름에 관심이 있는 고객
- 시연 전 테스트 데이터를 미리 확인해 둔 경우

설명 포인트:

- 공고 수집
- Go/No-Go 판단
- downstream 문서 생성
- 제안서/수행계획 연결 가능성

## 6. 대표 질문 대응 문구

### 이거 SaaS인가요?

`기본 권장 구조는 내부 설치형입니다. 웹으로 접속하지만, 고객사별로 독립 배포해서 운영할 수 있습니다.`

### 우리 회사 데이터가 다른 고객과 섞이나요?

`실도입 시에는 고객사 전용 환경으로 분리합니다. 도메인, 키, 데이터 볼륨, 운영 키를 분리하는 구조입니다.`

### 일반 AI 도구와 가장 큰 차이는 뭔가요?

`DecisionDoc AI는 대화형 답변보다 실제 문서 생성과 운영 흐름에 초점을 둡니다. 생성 후 승인, 공유, 내보내기, 프로젝트 이력 관리까지 이어집니다.`

### 지금 바로 우리 회사용으로도 만들 수 있나요?

`가능합니다. 현재 데모 환경은 공용 운영 환경이고, 실제 도입 시에는 귀사 전용 서브도메인과 전용 배포 환경으로 분리하는 방식을 권장합니다.`

## 7. 미팅 후 후속 액션

미팅이 끝나면 아래 중 하나로 바로 넘기는 것을 권장합니다.

- 제품 소개서 PDF 전달
- 내부 설치형 도입 설명서 전달
- 고객사 전용 환경 분리 배포 제안
- 파일럿 범위 정의 미팅 제안

## 8. 운영자 메모

- 외부 미팅에서는 `admin` 환경을 “데모/운영자 공용 환경”으로 설명할 것
- 고객 전용 환경은 실제 계약 또는 파일럿 범위 확정 후 별도 분리하는 구조로 안내할 것
- 기능 과장보다 배포 구조, 데이터 분리, 실제 문서 산출물 중심으로 설명할 것
